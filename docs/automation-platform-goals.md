# 자동매매 서비스 목표 체크리스트

## 문서 목적

- [ ] 회원제 자동매매 서비스 전환 전에 무엇을 먼저 끝내야 하는지 우선순위를 고정한다.
- [ ] 비용 폭탄, 권한 우회, API key 노출, 실거래 오작동을 출시 차단 조건으로 둔다.
- [ ] 첫 목표는 실거래 자동주문이 아니라 `로그인 + 베타 권한 + 전략 설정 + 모의 실행 + 주문의도 초안`까지로 제한한다.
- [ ] 기존 paper-trading 자동운용 탭은 숨기고, 사용자가 직접 세팅하는 신규 자동매매 탭을 별도로 만든다.
- [ ] v1은 국내/미국 주식 전략을 우선하고, 가상화폐 자동매매는 거래소별 API 정책과 위험 경계를 다시 잡은 뒤 후순위로 둔다.

## 성공 기준 요약

- [ ] 인증하지 않은 사용자는 자동매매 관련 API와 화면에 접근할 수 없다.
- [ ] 회원가입한 사용자 중에서도 베타 코드가 승인된 사용자만 자동매매 기능을 볼 수 있다.
- [ ] UI에서 숨긴 기능은 서버에서도 `403`으로 차단된다.
- [ ] 악의적인 반복 호출이 Vercel, Supabase, 외부 시세 API 비용을 과도하게 만들 수 없도록 공유 rate limit과 쿼터가 적용된다.
- [ ] 거래소/증권사 API key는 브라우저, localStorage, 일반 API 응답, 로그에 노출되지 않는다.
- [ ] 사용자는 차트에서 지지선/저항선을 드래그해 전략을 설정할 수 있다.
- [ ] 사용자는 지지선과 저항선을 각각 움직이거나, 박스권 세트로 함께 이동할 수 있다.
- [ ] 모든 전략은 손실 제한, 매수/매도 횟수 제한, 종료 조건을 반드시 가진다.
- [ ] v1에서는 실제 주문 제출 없이 모의 실행과 `OrderIntent` 초안까지만 생성한다.
- [ ] 실거래는 별도 승인 단계에서 `OrderIntent + RiskCheck + BrokerCredential + user ownership` 검증을 통과한 뒤에만 가능하다.

## 벤치마크 방향

- [ ] MagicSplit류 서비스의 핵심인 분할 매수/매도, 차수 관리, 예약 실행, 그룹 관리 흐름을 참고한다.
- [ ] 단순 복제 목표가 아니라 `차트에서 지지/저항선을 직접 잡고 전략과 위험 제한을 즉시 구성하는 UX`를 차별점으로 둔다.
- [ ] 사용자가 숫자 입력만으로 전략을 만드는 것이 아니라, 차트 조작 결과가 라더와 리스크 요약으로 바로 연결돼야 한다.
- [ ] 초보 사용자는 프리셋으로 시작하고, 숙련 사용자는 차수/조건/종료 규칙을 세부 조정할 수 있어야 한다.
- [ ] 모든 자동화 표현은 수익 보장처럼 보이지 않게 하고, 사용자가 설정한 조건을 실행 보조하는 도구로 표현한다.

## 0. 선행 게이트: 비용/남용 방어

### 0.1 API 위험도 분류

- [ ] 모든 API route를 `public-read`, `auth-read`, `auth-write`, `expensive-analysis`, `automation`, `broker-secret` 범위로 분류한다.
- [ ] 각 endpoint에 예상 비용 요인을 적는다: Supabase read/write, 외부 시세 호출, 커뮤니티 수집, 백테스트 반복, 브로커 검증 호출.
- [ ] 비싼 endpoint는 기본적으로 인증 사용자만 호출 가능하게 한다.
- [ ] public endpoint는 데이터 범위와 기간을 더 작게 제한한다.
- [ ] 자동매매/브로커/전략 저장 endpoint는 anon 접근을 전부 차단한다.

### 0.2 공유 rate limit

- [ ] Vercel serverless 인스턴스별 메모리 제한이 아니라 공유 저장소 기반 limiter를 둔다.
- [ ] 후보는 Upstash Redis, Supabase Postgres counter, Vercel KV 중 하나로 결정한다.
- [ ] 최소 기준은 `IP + user_id + endpoint_scope` 조합으로 제한한다.
- [ ] 미인증 사용자는 IP 기준 제한을 적용한다.
- [ ] 인증 사용자는 user_id 기준 제한을 IP 기준보다 우선 적용한다.
- [ ] 베타 자동매매 사용자는 별도 더 엄격한 automation quota를 적용한다.
- [ ] 제한 초과 시 `429`와 retry 가능 시간을 반환한다.
- [ ] limiter 장애 시 expensive endpoint는 fail-closed, 일반 read endpoint는 제한적으로 fail-open할지 정책을 문서화한다.

### 0.3 요청 범위 제한

- [ ] `days`, `start`, `end`, `limit`, `symbols`는 endpoint별 최대값을 가진다.
- [ ] 일반 분석 기간 상한은 3년을 넘지 않는다.
- [ ] 리더 스캔 기간 상한은 2년을 넘지 않는다.
- [ ] crypto backtest 기간 상한은 1년을 넘지 않는다.
- [ ] 자동매매 시뮬레이션은 v1에서 단일 종목, 단일 전략, 제한된 캔들 수만 허용한다.
- [ ] 과도한 범위 요청은 자동 축소하지 말고 `400`으로 거부한다.
- [ ] 외부 fetch에는 AbortSignal timeout을 적용한다.
- [ ] 응답 body 크기 예산을 둔다.
- [ ] 병렬 외부 호출은 bounded concurrency를 적용한다.

### 0.4 비용 관측과 차단

- [ ] rate limit hit, 4xx, 5xx, 외부 API timeout을 구조화 로그로 남긴다.
- [ ] abuse 의심 이벤트를 저장할 `rate_limit_events` 또는 동등한 로그 저장소를 둔다.
- [ ] IP/user blocklist를 수동으로 적용할 수 있어야 한다.
- [ ] Vercel 사용량 알림을 설정한다.
- [ ] Supabase usage/billing 알림을 설정한다.
- [ ] 외부 시세/브로커 API 호출량 알림을 설정한다.
- [ ] 운영자가 특정 사용자 자동매매 권한을 즉시 끌 수 있어야 한다.

## 1. 로그인/회원가입

- [ ] Supabase Auth를 사용해 이메일 회원가입과 로그인을 먼저 제공한다.
- [ ] 인증 세션은 서버 API에서 Supabase `getUser` 기준으로 검증한다.
- [ ] 클라이언트가 전달한 user_id를 신뢰하지 않는다.
- [ ] `profiles` row는 회원가입 후 생성되거나 첫 로그인 시 보정된다.
- [ ] 로그아웃 후 자동매매 API 호출은 `401`을 반환한다.
- [ ] 로그인 화면은 자동매매 탭 접근 시 자연스럽게 연결된다.
- [ ] 로그인하지 않아도 볼 수 있는 기존 공개 분석 기능과 로그인 전용 기능을 명확히 분리한다.
- [ ] 인증 실패 메시지는 계정 존재 여부를 과도하게 노출하지 않는다.
- [ ] Supabase `service_role` key는 서버 전용 env에만 둔다.
- [ ] 브라우저에 노출되는 env는 publishable key 수준으로 제한한다.

## 2. 베타 권한 제어

### 2.1 정책

- [ ] 회원가입만으로 자동매매 기능이 열리지 않는다.
- [ ] 베타 코드를 등록한 사용자에게만 자동매매 feature flag를 부여한다.
- [ ] 베타 권한은 회수 가능해야 한다.
- [ ] 코드가 유출되면 해당 코드를 비활성화할 수 있어야 한다.
- [ ] 베타 권한은 UI 숨김이 아니라 API 권한으로도 검증한다.

### 2.2 데이터 모델

- [ ] `beta_invite_codes` 테이블을 설계한다.
- [ ] code 원문은 저장하지 않고 hash를 저장한다.
- [ ] 코드별 최대 사용 횟수와 만료일을 가진다.
- [ ] `user_feature_access` 테이블을 설계한다.
- [ ] `user_feature_access`는 `user_id`, `feature`, `status`, `granted_at`, `revoked_at`을 가진다.
- [ ] feature 예시는 `automation_beta`, `live_trading`, `broker_credentials`로 분리한다.
- [ ] RLS는 사용자 본인 row 조회만 허용하고, 권한 부여/회수는 서버 전용으로 제한한다.

### 2.3 API

- [ ] `GET /api/me/features`는 현재 사용자의 feature flag만 반환한다.
- [ ] `POST /api/beta/redeem-code`는 로그인 사용자만 호출 가능하다.
- [ ] 잘못된 코드, 만료 코드, 사용 횟수 초과 코드는 동일하게 안전한 실패 메시지를 반환한다.
- [ ] 자동매매 관련 API는 feature flag가 없으면 `403`을 반환한다.
- [ ] feature flag 판정은 클라이언트 상태가 아니라 서버 조회 결과를 기준으로 한다.

## 3. 기존 자동매매 탭 숨김

- [ ] 현재 paper-trading 자동운용 탭은 기본 navigation에서 제거하거나 feature flag 뒤로 숨긴다.
- [ ] 숨김 처리 후에도 기존 API는 인증과 소유권 검사를 유지한다.
- [ ] 기존 paper-trading API가 사용자의 `account`, `positions`, `today` 주입을 받지 않도록 유지한다.
- [ ] 기존 기능은 내부 검증/운영자용으로만 남길지, 완전히 제거할지 별도 결정한다.
- [ ] 사용자가 새 자동매매 탭과 기존 paper-trading 기능을 혼동하지 않게 라우팅과 이름을 분리한다.

## 4. 신규 자동매매 탭 정보 구조

### 4.1 기본 화면

- [ ] 첫 화면은 전략 설정 화면이어야 하며 마케팅/설명 랜딩이 아니어야 한다.
- [ ] 모바일과 PC 모두 차트 조작이 우선 보이도록 구성한다.
- [ ] 상단에는 종목, 시장, 전략 프리셋, 모의/실거래 상태를 배치한다.
- [ ] 실거래가 비활성화된 v1에서는 화면에 명확히 `모의 실행` 상태를 표시한다.
- [ ] 기존 분석 탭과 동일한 종목 검색 흐름을 재사용한다.
- [ ] 자동매매 탭은 베타 권한이 없으면 navigation에 노출하지 않는다.

### 4.2 핵심 패널

- [ ] 차트 패널: 캔들, 지지선, 저항선, 박스권, 주문 후보 마커를 표시한다.
- [ ] 전략 패널: 프리셋, 라더, 진입 조건, 청산 조건을 편집한다.
- [ ] 리스크 패널: 최대 손실, 최대 매수 횟수, 최대 보유 금액, 일일 중지 조건을 표시한다.
- [ ] 시뮬레이션 패널: 예상 체결, 손익 시나리오, 주문의도 초안을 표시한다.
- [ ] 로그 패널: 최근 모의 실행 결과와 차단 사유를 표시한다.

## 5. 차트 기반 전략 세팅

### 5.1 공통 차트 조작

- [ ] 지지선과 저항선은 차트 위에서 드래그할 수 있다.
- [ ] 가격 입력 필드와 차트 드래그 결과가 양방향으로 동기화된다.
- [ ] 드래그 중에는 예상 매수/매도 가격과 손익률을 즉시 갱신한다.
- [ ] 모바일에서는 손가락 조작이 가능하도록 hit area를 충분히 크게 둔다.
- [ ] 확대/축소, 캔들 이동, 선 이동 동작이 충돌하지 않게 interaction mode를 둔다.
- [ ] undo/redo 또는 변경 전 복원 기능을 둔다.
- [ ] 저장 전 변경사항과 저장된 전략 상태를 구분해서 표시한다.
- [ ] 저장된 전략은 `strategy_configs`에 사용자 소유 row로 저장한다.

### 5.2 지지반등 프리셋

- [ ] v1 기본 프리셋은 `지지반등`으로 둔다.
- [ ] 사용자는 지지선, 진입 허용 오차, 분할 매수 차수, 목표가, 손절가를 설정한다.
- [ ] 지지선 근접 후 반등 확인 조건을 선택할 수 있다.
- [ ] 반등 확인 조건 예시는 양봉 전환, 거래량 증가, 직전 저점 이탈 없음으로 둔다.
- [ ] 지지선 이탈 시 신규 매수를 중지한다.
- [ ] 손절가 도달 시 미체결 주문 취소와 보유분 청산 의도를 생성한다.
- [ ] 반등 실패 횟수 제한을 설정할 수 있다.

### 5.3 박스권 프리셋

- [ ] 박스권은 지지선과 저항선을 한 세트로 관리한다.
- [ ] 지지선만 드래그하면 박스 하단만 이동한다.
- [ ] 저항선만 드래그하면 박스 상단만 이동한다.
- [ ] 박스 몸통을 드래그하면 지지선과 저항선이 간격을 유지한 채 함께 이동한다.
- [ ] 세트 이동 중에는 박스 폭, 중앙값, 예상 손익률을 유지하거나 재계산하는 기준을 명확히 한다.
- [ ] 사용자는 박스권 매수 구간과 매도 구간을 여러 단계로 나눌 수 있다.
- [ ] 박스권 하단 이탈 시 전략 중지 또는 손절 실행 조건을 선택할 수 있다.
- [ ] 박스권 상단 돌파 시 추세전환 관찰, 일부 청산, 전체 청산 중 하나를 선택할 수 있다.
- [ ] 박스권이 너무 좁아 수수료/슬리피지를 이기기 어렵다면 저장을 막거나 경고한다.

### 5.4 후순위 프리셋

- [ ] 삼각수렴 예측은 v1 핵심 이후로 둔다.
- [ ] 삼각수렴은 상단 추세선, 하단 추세선, 수렴 종료 예상 시점이 필요하다.
- [ ] 돌파/이탈 방향이 확정되기 전 실거래 자동주문은 기본 차단한다.
- [ ] 단타 스캘핑 프리셋은 수수료, 슬리피지, 호가 단위, API 지연을 반영할 수 있을 때 추가한다.

## 6. 라더, 제한, 종료 조건

### 6.1 분할 주문 라더

- [ ] 사용자는 매수 차수 수를 제한할 수 있다.
- [ ] 각 차수는 가격, 금액 또는 비중, 조건, 만료 시간을 가진다.
- [ ] 차수별 중복 체결 방지 key를 둔다.
- [ ] 이미 체결된 차수는 재실행 시 다시 주문되지 않는다.
- [ ] 라더 총액이 사용자 설정 최대 보유 금액을 넘으면 저장을 막는다.
- [ ] 차수 가격이 현재가와 너무 가까우면 경고한다.

### 6.2 매수/매도 횟수 제한

- [ ] 일일 최대 매수 횟수를 설정할 수 있다.
- [ ] 일일 최대 매도 횟수를 설정할 수 있다.
- [ ] 전략 전체 최대 매수 횟수를 설정할 수 있다.
- [ ] 종목별, 전략별, 계정별 제한을 분리할 수 있어야 한다.
- [ ] 제한 초과 시 신규 주문의도는 `blocked` 상태로 기록된다.

### 6.3 종료 조건

- [ ] 목표 수익률 도달 시 전략 종료를 선택할 수 있다.
- [ ] 최대 손실률 도달 시 전략 종료를 선택할 수 있다.
- [ ] 최대 보유 시간 초과 시 전략 종료를 선택할 수 있다.
- [ ] 연속 실패 횟수 초과 시 전략 종료를 선택할 수 있다.
- [ ] 사용자가 말한 `주식회수`는 `미체결 주문 취소 + 보유 수량 청산 의도 생성 + 전략 비활성화`로 정의한다.
- [ ] v1 모의 실행에서는 실제 취소/청산 주문 대신 취소/청산 `OrderIntent` 초안을 만든다.
- [ ] 종료 조건 발동 시 사용자가 원인을 확인할 수 있도록 `execution_logs`에 남긴다.

## 7. 모의 실행과 주문의도

- [ ] v1 자동매매는 실제 브로커 주문을 제출하지 않는다.
- [ ] 전략 실행 결과는 `OrderIntent` 초안으로만 저장한다.
- [ ] `OrderIntent` 생성 전 `RiskCheck`를 반드시 실행한다.
- [ ] RiskCheck 실패 시 주문의도는 `blocked`가 되거나 생성되지 않는다.
- [ ] 주문의도에는 user_id, strategy_config_id, market, symbol, side, order_type, quantity/notional, limit_price, reason이 포함된다.
- [ ] 주문의도 응답에는 로컬 경로, 서버 내부 상태, secret 값이 포함되지 않는다.
- [ ] 시뮬레이션 결과는 현재가, 지지/저항선, 라더 조건, 제한 조건을 기준으로 재현 가능해야 한다.
- [ ] 동일 입력의 시뮬레이션은 동일 결과를 내야 한다.

## 8. Supabase 데이터와 RLS

### 8.1 기존 기반 확인

- [ ] `profiles`는 사용자 기본 정보를 저장한다.
- [ ] `portfolios`는 사용자별 보유 종목을 저장한다.
- [ ] `paper_trading_states`는 사용자별 paper 상태를 저장한다.
- [ ] `strategy_configs`는 사용자별 전략 설정을 저장한다.
- [ ] `broker_credentials`는 서버 전용 암호화 credential 저장소로만 사용한다.
- [ ] `order_intents`는 주문 후보와 상태를 저장한다.
- [ ] `risk_checks`는 주문 후보별 리스크 판정 결과를 저장한다.
- [ ] `execution_logs`는 실행/차단/오류 이벤트를 저장한다.

### 8.2 추가 필요 테이블

- [ ] `beta_invite_codes`를 추가한다.
- [ ] `user_feature_access`를 추가한다.
- [ ] `rate_limit_events` 또는 abuse log 저장소를 추가한다.
- [ ] 필요하면 `strategy_simulation_runs`를 추가해 모의 실행 결과를 재조회 가능하게 한다.
- [ ] 필요하면 `strategy_templates`를 추가해 기본 프리셋을 서버에서 관리한다.

### 8.3 RLS 기준

- [ ] 모든 user-owned table은 `user_id = auth.uid()` 정책을 가진다.
- [ ] UPDATE 정책에는 SELECT 정책도 함께 존재해야 한다.
- [ ] `broker_credentials`는 authenticated 직접 접근을 허용하지 않는다.
- [ ] credential 등록/삭제/상태 조회는 서버 route에서 service role로만 처리한다.
- [ ] RLS 정책은 anon, authenticated, service_role 관점으로 테스트한다.
- [ ] 권한 정보는 user-editable metadata가 아니라 서버 관리 테이블 또는 app_metadata를 사용한다.

## 9. 브로커/증권사 API 준비

- [x] 토스증권 Open API 출시 문서를 기준으로 계좌, 보유, 시세, 주문 가능 정보, 주문 조회, 장 캘린더 연동을 구현 대상으로 둔다.
- [x] 토스 credential 등록은 자동매매 베타 권한과 분리하고, `broker_credentials` feature로 개인화 기능을 연다.
- [ ] 국내/미국 주식 실거래 주문 제출은 Toss API 제공 범위와 서비스 안전장치를 별도 검증한 뒤 활성화한다.
- [x] Upbit/Bithumb은 코인 설정과 거래소별 주문 어댑터로 분리 구현한다.
- [x] 전략 평가·RiskCheck·kill switch는 공유하되 Toss/Upbit/Bithumb 주문 규격과 credential 경계는 분리한다.
- [x] 브로커 공통 interface에서 주문 제출과 주문 취소를 제공하고, 거래소별 계좌·주문 가능 점검은 별도 endpoint로 노출한다.
- [x] 입출금 API는 사용하지 않는다.
- [x] credential list API는 등록 여부, broker, masked_identifier, status만 반환한다.
- [x] secret value read endpoint는 만들지 않는다.

## 10. 클라이언트 저장소 기준

- [ ] localStorage에는 theme, 접힘 상태, 마지막 탭 같은 민감하지 않은 preference만 둔다.
- [ ] 보유 수량, 평단, 메모, 전략 설정은 Supabase user-owned table로 이동한다.
- [ ] 브로커 credential, access token, refresh token, secret key는 localStorage/sessionStorage에 저장하지 않는다.
- [ ] 자동매매 설정 draft가 브라우저에 임시 저장될 경우 민감 값이 포함되지 않아야 한다.
- [ ] 로그아웃 시 사용자별 UI cache와 draft를 정리한다.
